# 20260715_151 — 반출 위생: 운영 DB 사본에 실린 세션 토큰

> 성격: **보안 수정.** [devlog 150](20260715_150_hourly-integrity-gate.md)(백업 레인)의 자매편 —
> 150 이 *"백업이 복원 가능한가"* 였다면 이건 *"백업이 어디까지 나가는가"* 다.
> 계기: 사용자 — *"운영서버의 DB 를 cron 으로 복사해 와서 테스트서버의 DB 로 사용하고 있잖아. 보안상의 문제는 없어?"*
> 릴리스 없음(호스트 스크립트 `scripts/sync-cdgts-db.sh` 만 — 이미지 미탑재 경로).

## 1. 먼저, 대부분은 괜찮았다

| 항목 | 실측 | 판정 |
|---|---|---|
| 제3자 개인정보(PII) | `auth_user` = **본인 admin 1개**, 학자 사용자 0 · `accounts_profile`·`authtoken_token`·`socialaccount` 테이블 없음 | 🟢 유출될 남의 정보가 없다 |
| 비밀번호 해시 | `pbkdf2_sha256$**1000000**$…` (Django 5.2 기본) | 🟢 강함 |
| 테스트서버 노출 | `tailscale serve` = **tailnet only** (funnel 아님) | 🟢 공개 인터넷 미도달 |
| 테스트 `DEBUG` | `False` | 🟢 |
| `SECRET_KEY` | prod ≠ test | 🟢 …인 줄 알았다(§3) |

**PII** = Personally Identifiable Information(개인식별정보; 개인정보보호법의 "개인정보"). 이 프로젝트엔 아직 없다 —
학자 사용자가 붙으면 이 표의 첫 줄이 바뀌고, 그때 이 문서의 결론도 세진다.

## 2. 문제: `session_key` 는 **bearer 토큰 그 자체**다

`django_session` 에 **아직 만료되지 않은 세션 3개**(만료 7/17·7/22·7/25)가 매일 복사되고 있었다.

Django 세션에서 쿠키에 담기는 값은 `session_key` **문자열 하나**다. 서버는 그걸로 자기 DB 행을 찾고
`session_data` 를 자기 `SECRET_KEY` 로 디코드한다. 그래서 공격은 이렇게 된다:

```
사본에서 session_key 를 읽는다
        ↓  쿠키에 담아
운영서버(cdgts.paleobytes.info)에 제시
        ↓  운영이 자기 DB 에서 그 행을 찾고
        ↓  자기 SECRET_KEY 로 디코드 → 성공
admin(superuser) 로 로그인됨
```

## 3. `SECRET_KEY` 가 달라도 안 막힌다 — 여기가 함정

*"prod 와 test 의 SECRET_KEY 가 다르니 안전하다"* 는 **틀렸다.** 키 차이가 막는 건
"**테스트서버가** 운영 `session_data` 를 해독하는 것"뿐이다. 위 공격은 **해독을 하지 않는다** —
훔친 키를 **운영에** 되돌려주면 운영이 자기 키로 알아서 푼다.

> **해시는 뚫어야 쓰지만, 세션 키는 그냥 쓰면 된다.** 1,000,000회 PBKDF2 옆에
> 평문 등가의 bearer 토큰이 나란히 실려 나가고 있었다.

그리고 대칭적으로: **그 세션들은 테스트서버에선 애초에 무용지물**이었다(SECRET_KEY 불일치 → 서명 검증 실패 →
빈 세션). 즉 이 행들은 **테스트엔 쓸모없고 운영엔 위험한 순수 손해**였다. 지워서 잃는 게 없다.

## 4. 노출면 — 사본이 29벌이었다

```
m710q /srv/cdGTS/db          1   (라이브 테스트 DB)
m710q ~/backups/current      1
m710q ~/backups/db_history  13
NAS   db_history            13   ← drwxrwxrwx (0777)
NAS   current                1   ← drwxrwxrwx (0777)
```

NAS 디렉터리가 **0777**이고 보존이 **90일**이다. 실측: **7월 3일자 사본(12일 전)에도 아직 유효한 세션**이 있었다.
스냅샷은 시점 사진이라 **토큰이 만료될 때까지 모든 세대에 복제된다.**

## 5. 처방 — 반출 위생(egress sanitize)

**신뢰 경계를 넘기 직전에** 지운다. `sync-cdgts-db.sh` 는 이미 운영에서 `src.backup(dst)` 로 임시 스냅샷을
만들고 그것만 scp 하므로, 그 **임시 사본**에 대고 한 줄 더 하면 토큰이 **운영을 떠나지 않는다**:

```
prod 라이브 DB ──.backup()──▶ /srv/cdGTS/.db_sync_snapshot ──scp──▶ m710q ──▶ NAS
   (읽기만)                      ↑ 여기서 DELETE
```

```python
src.backup(dst)
dst.execute("PRAGMA secure_delete=ON")                   # 빌드 기본값에 기대지 않는다(§9)
n = dst.execute("DELETE FROM django_session").rowcount   # dst = 반출될 사본. src 는 읽기만.
dst.commit()
dst.execute("VACUUM")                                    # ★ 필수 — 상속된 free page 잔류(§9). commit 뒤에.
dst.execute("PRAGMA journal_mode=DELETE")                # 아카이브 = 단일 파일(150 §10 동형)
```

위생 실패 시 **`sys.exit(1)` 로 반출 자체를 중단**한다 — 검사에 실패한 스냅샷을 내보내느니 sync 를 실패시킨다
(150 의 *"검증 실패하면 채택하지 않는다"* 와 같은 형태).

> ⚠️ **이 블록의 `secure_delete`·`VACUUM` 두 줄은 초판에 없었다** — `DELETE` 만으로는 구조적으로 부족하다는 걸
> 나중에야 실측으로 알았다. 경위와 근거는 **§9**.

### 왜 세션은 지우고 해시는 남기나

| | 지우나 | 이유 |
|---|---|---|
| `django_session` | **지움** | 평문 등가 bearer 토큰. 테스트엔 무용지물(§3) → 순수 손해 |
| `auth_user.password` | **남김** | 1,000,000회 PBKDF2 로 강하고, 지우면 **테스트서버 로그인이 막힌다**. "그냥 쓸 수 있는" 값이 아니라 성질이 다르다 |

## 6. 사용자 질문이 잡아낸 내 모호한 문장

> *"운영 쪽에서 세션을 지우면 운영 쪽에 연결되어 있는 사람은 매일 로긴을 새로 해야 하는 거야?"*

**아니다 — 그리고 그렇게 읽히게 쓴 건 내 잘못이다.** 초안에 *"운영 쪽에서 `.backup()` 직후 세션을 지우면"*
이라고 썼는데, "운영 쪽에서"는 *운영 호스트에서 실행된다*는 뜻이었지 *운영 DB 를 지운다*는 뜻이 아니었다.

**라이브 DB 는 읽기만 한다.** 실측으로 못 박음 — sync 실행 전후 운영 세션 **3 → 3**(§8).
코드 주석에도 이 구분을 명시해 뒀다(⚠️ 대상은 dst 뿐).

> 이 오해는 위험한 종류였다: 그대로 승인됐으면 **사용자는 매일 로그아웃되는 줄 알면서** 수락했을 것이고,
> 반대로 내가 정말 라이브를 지우는 코드를 짰다면 아무도 못 잡았을 것이다. **"어느 DB냐"가 이 변경의 전부**인데
> 그걸 대명사에 맡겼다.

## 7. 소급 정리 — 새 사본만 깨끗해서는 안 닫힌다

기존 26+벌엔 여전히 살아있는 토큰이 있었다(수정은 *앞으로*의 사본만 고친다). 전수 정리:

```
소급 정리: 사본 28벌에서 세션 44행 제거(그중 아직 유효했던 것 44개)
```

**44개 전부가 아직 유효**했다 — 만료된 게 하나도 없었다는 건 노출이 이론이 아니었다는 뜻이다.
`VACUUM` 도 함께 걸었다(~~`DELETE` 만으로는 free page 에 토큰 문자열이 그대로 남으므로~~ — **이 근거는 틀렸다. §9 참조.**
결론은 유지되지만 이유가 다르고, 더 중요하게는 **상시 반출 경로엔 그 `VACUUM` 을 안 넣었다** — 그것도 §9).

## 8. 검증

| 확인 | 결과 |
|---|---|
| ★ **운영 라이브 세션**(sync 전/후) | **3 → 3** — 아무도 로그아웃 안 됨 |
| 운영 임시 스냅샷 잔재 | 0 |
| 반출 사본 4종 표본(/srv·current·history·NAS) | 세션 **0** · `auth_user` **1 보존** · `journal_mode=delete` |
| 전 사본 전수 재확인 | **세션 0 ✓** |
| 격리 테스트(합성 WAL DB) | 스냅샷 세션 0 / 라이브 소스 3 온전 / 부산물 0 |
| **`VACUUM` 재구축 증거**(§9 수정 후 재실행) | 반출본 **`freelist_count = 0`** · `integrity ok` |

## 9. `VACUUM` — 틀린 근거로 옳은 코드를 지키다가, 정작 필요한 자리엔 빠뜨렸다

> 계기: 사용자 — *"session 정보 삭제할 때 vacuum 관련된 내용 체크해봐."* 확인해보니 **버그 1건 + 틀린 근거 1건**이 나왔다.

### 9.1 초판이 적은 근거는 틀렸다

§7 에 *"`DELETE` 만으로는 free page 에 토큰 문자열이 그대로 남는다"* 라고 **확인 없이 단정**했다. 실측:

```
PRAGMA secure_delete 기본값 = 1 (ON)   ← m710q 3.45.1 · dolfinid 3.45.1 · 컨테이너 3.46.1 전부
  DELETE 만            → 파일에 남은 토큰   0개
  secure_delete=OFF 로 두면 → 남은 토큰 200개 (그제야 통설대로 grep 에 잡힌다)
```

`secure_delete=ON` 이면 sqlite 가 지운 자리를 **0 으로 덮는다.** 즉 통설은 **빌드 의존**이고, 우리 빌드에선 거짓이었다.
(병렬 fcmanager 라운드도 같은 날 독립적으로 이걸 반증하고 계약을 정정했다.)

### 9.2 그런데 진짜 이유는 더 강했고, 그건 **오늘 당장** 성립한다

`secure_delete` 는 **지금 지우는 행**만 덮는다. 그런데 `backup()` 은 소스를 **페이지 단위로 복사해 free page 까지
그대로 옮긴다.** 운영 DB 에서 *과거에* 지워진 세션(로그아웃·만료 정리·로그인 시 세션 회전)의 잔류가 스냅샷에
딸려오는데, **그건 이미 행이 아니라 `DELETE` 가 구조적으로 닿지 못한다**:

```
소스 free page 에 200개
   → backup() 직후 스냅샷            200개   ← backup 은 free page 도 복사한다
   → secure_delete=ON + DELETE 후    200개   ← DELETE 가 못 닿는다
   → VACUUM 후                         0개   ← 재구축만이 해소
```

**그래서 노출은 위생을 *실행하는* 빌드가 아니라 그 DB 를 평생 건드린 *모든 writer* 의 빌드에 걸린다.**
그리고 실제 writer 는 호스트가 아니라 **컨테이너**다(cdGTS 실측: 컨테이너 3.46.1 / 호스트 3.45.1 — 마침 둘 다
`secure_delete=1` 이라 오늘 실질 잔류는 0, prod `freelist_count`=1 페이지도 0 으로 덮여 있다). **그 이력은 되짚을 수 없다.**

> `VACUUM` 은 "빌드가 바뀔 때를 대비한 보험"이 아니라 **이력을 수선하는 유일한 수단**이다.
> → 계약 §백업 레인에 *"`PRAGMA secure_delete=ON` 으로 `VACUUM` 을 대체할 수 없다"* 로 못 박았다.

### 9.3 정작 버그는 따로 있었다 — 상시 경로에 `VACUUM` 이 없었다

초판은 **소급 정리(§7)에만** `VACUUM` 을 넣고 **상시 반출 경로(§5)엔 빠뜨렸다.** 즉 오늘 이후의 모든 사본이
상속 잔류를 그대로 내보낼 수 있었다. 고쳤다 — `secure_delete=ON` 명시 + `VACUUM`(⚠️ 열린 트랜잭션이면
`cannot VACUUM - SQL statements in progress` 로 실패하니 **`commit()` 뒤에**).

**검증**(실제 sync 재실행): `sanitize: django_session 3행 제거 + VACUUM(상속 잔류 포함 재구축)` ·
반출본 **`freelist_count=0`**(= 파일 재작성 증거) · 세션 0 · `integrity ok` · ★ **운영 라이브 세션 3→3**.

### 9.4 교훈

- **틀린 근거로 옳은 코드를 지키면 다음 사람이 그 근거를 믿는다.** 그리고 그 근거를 믿으면 자연스러운 최적화가
  *"`secure_delete=ON` 명시하면 `VACUUM` 은 빼도 되겠네"* 인데 — **그게 정확히 틀린 결론**이다. 계약에 이 오독을
  명시적으로 막아뒀다(fsis 이식 작업 중이라 지금 필요했다).
- **회귀 테스트가 공허할 수 있다**(fcmanager 라운드가 같은 날 확인): 바이트 `grep` 단언은 `secure_delete` 가
  대신 해줘서 **`VACUUM` 을 빼도 통과**한다. `freelist_count == 0` 을 보되 **픽스처가 페이지를 여러 장 쓰게** 할 것
  — 세션 3행은 한 페이지라 `DELETE` 해도 루트가 남아 freelist 가 0 이다. [150 §10](20260715_150_hourly-integrity-gate.md)
  의 WAL 픽스처와 **같은 실패**(재현 조건이 픽스처에 없으면 단언은 장식).
- **오늘 다섯 번째 "확인 없이 단정"** 이다(149 오귀인 · 150 §10 오귀인 · 갱신 이력 "무손실" 오기 · RETAIN 24
  "튜닝 값이라 보편성 없다" · 그리고 이것). 매번 그럴듯한 메커니즘을 찾자마자 멈췄다. **그럴듯함이 확인을
  대신할 수 있다는 착각**이 이 라운드의 진짜 반복 결함이다.

## 10. 남은 것

- **NAS `drwxrwxrwx`(0777)** — 이번 수정은 *무엇을 내보내는가*를 고쳤지 *어디에 두는가*는 그대로다.
  토큰은 빠졌지만 강한 해시·그래프 데이터는 여전히 world-readable 디렉터리에 90일 산다. 별건으로 검토.
- ~~일반화 후보~~ → **계약 §백업 레인 MUST 로 승격 완료**(2026-07-15). 미뤄뒀던 적용 범위 확인의 결과가 승격 근거가 됐다:
  **운영 DB 를 호스트 밖으로 반출하는 건 3/5**(cdGTS·fcmanager·fsis2026 — 전부 NAS `drwxrwxrwx`), paleonews·naverland 는
  prod 가 빌드 호스트 자신이라 해당 없음. ⚠️ **fsis2026 이 훨씬 심각하다** — NAS 최신 사본 실측: **세션 184(유효 8)·
  사용자 60(이메일 보유 33)·사본 87벌**. cdGTS(본인 계정 1개)와 달리 **제3자 PII + 실이용자를 사칭 가능한 유효 토큰**이고,
  계약이 스스로 *"사용자 신뢰성이 걸린 건 kofhin(fsis2026) 정도뿐"* 이라 적어둔 프로젝트다. fcmanager 는 적용 완료(0.6.25,
  62벌·1,992행·유효 16개 소급 정리), fsis2026 은 작업 중.
- **`MAILTO` 미설정**(150 §11 과 공유) — sync 는 배포 경로가 아니라 **smoke 가 못 잡는다.** 위생이 조용히
  깨지면(예: 테이블명 변경) `sys.exit(1)` 이 cron 로그에만 남는다. 150 의 센티넬→healthz 같은 노출 채널이 없다.
