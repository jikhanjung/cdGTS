# 전역 vs 경계별 버전 — 열린 질문

*[English](versioning-global-vs-per-boundary_en.md) · 한국어*

> 상태: **검토 노트. 결정 아님.** [boundary-gateway-schema.md](boundary-gateway-schema.md) §4의
> "전역 vs 경계별 버전" 항목을 펼친 것. 여기서는 문제를 분해하고 **열린 질문으로 정의**만 한다.
>
> **[일부 구현됨]** 전역 릴리스 = 고정 레코드 집합(bake) + 정합성 게이트로 구현됨(`releases` 앱, bake/narrate).
> 아래 열린 질문 일부는 이 구현으로 해소; 경계별 독립 버전 매니페스트는 여전히 미완(열린 질문).

## 1. 문제 (한 줄)

소프트웨어의 고전 문제인 **"컴포넌트 버전 vs 릴리스 락파일(lockfile)"** 과 같다 — 패키지마다 버전이 있고,
그것들을 한 순간에 고정한 락파일이 따로 있듯이. 다만 지질학은 순수 소프트웨어엔 없는 비틀기를 얹는다:
경계들 사이에 지켜야 할 **정합성 제약(coherence)** 때문에, 전역 릴리스가 "각 경계의 최신값을 그냥 모은 것"이
될 수 없다.

## 2. 하위 문제

- **A. 케이던스 불일치 (cdGTS의 존재 이유).** 과학은 경계별로 **비동기·연속** 갱신(P–T 2014, 캄브리아 base
  논쟁 중, Ediacaran 2004)되는데 출판(ICC/GTS)은 **전역·주기적**이다. "경계별 갱신이 10년짜리 전역 릴리스를
  기다리면 안 된다"가 논지이므로 경계별 버전은 필수. 하지만 인용·권위는 안정된 전역 스냅샷을 요구 → 둘 다 필요.

- **B. 전역 릴리스가 *무엇*인가.** ① 모든 경계 레코드의 **복사 스냅샷** vs ② `{boundary_id → boundary_version}`
  **매니페스트(락파일)** + 공유 노드 버전 + 모델 선택. ②는 git의 (불변 blob들 + 이를 가리키는 tree/tag)와 같은 모양.

- **C. 정합성 (진짜 어려운 곳).** 경계들은 독립이 아니다 → §3에서 별도로.

- **D. 경계에 안 걸리는 변경.**
  - *공유 노드 재보정:* 붕괴상수·tracer 하나가 수십 경계에 물려 있어(P–T 케이스), 바뀌면 **동시에** 움직인다.
    본질적으로 전역 이벤트인데 경계별 버전만으론 어색.
  - *토폴로지/명명 변경:* stage 분할, GSSA→GSSP 전환은 경계의 **집합 자체**(추가/삭제/개명)를 바꾼다.
    경계별 버전은 "정체성 고정 + 값 변경"만 다룸 → 전역 버전이 **집합·위상**까지 버전해야.

- **E. 샌드박스 = 베이스라인 + 오버라이드.** "이 U-Pb 하나 넣으면?"은 전역 베이스라인 위에 얹은 **경계별 델타**.
  "ICC-2024/12를 쓰되 base-cambrian만 내 실험 버전으로" = 락파일 + 로컬 오버라이드 패턴.

- **F. 인용·권위의 비대칭.** 비준은 **경계별**(P–T 2001, 캄브리아 1992, Ediacaran 2004), 출판은 **전역**.
  `status.level: ratified`는 경계에 살고, 전역 릴리스는 비준된 경계를 큐레이션해 묶는 거버넌스 행위.
  재현하려면 인용이 **경계 버전 + 전역 릴리스 맥락**을 둘 다 가리켜야.

## 3. 정합성(C)에 대한 현실 감각

경계들 사이엔 최소 세 제약이 있다:

- **단조 순서(monotonic ordering):** 차트에서 base(젊은 단위) 연대 < base(오래된 단위) 연대. 경계별로 따로
  업데이트해 "각자 최신값"을 모으면 원리적으로 **역전**이 가능.
- **지속시간(duration):** stage 길이 = 두 경계 연대의 차. 서로 다른 버전에서 뽑은 두 경계를 그냥 빼면 잘못됨.
- **상관된 불확실성:** 공유 노드(붕괴상수·tracer)가 여러 경계에 물리면 오차가 부분적으로 **상관**된다. 지속시간의
  불확실성은 단순 √합보다 작을 수 있어, 전역 릴리스가 두 개의 `±` 값만이 아니라 **상관 구조 자체**를 알아야 한다.

> **현실 단서(중요).** ICC에 올라갈 만큼 널리 쓰이는 대표 경계들은 서로 **충분히 떨어져 있어** 실제로 시간
> 역전이 일어날 만큼 가깝지 않다. 즉 단조 순서 위반은 대표 경계에선 대체로 **이론적**이다. 그럼에도 단조 순서는
> 모델이 **알고 있어야 할 불변식**이며, 문제가 실제로 무는 곳은 **세밀하게 분할된 구간**(예: 촘촘한 stage들)이나
> **2σ 구간이 겹치는 경우**, 그리고 위의 지속시간·상관오차 계산이다. → 그래서 전역 릴리스는 "최신값 모으기"가
> 아니라 최소한 **단조 순서 확인**을, 이상적으로는 상관오차까지 다루는 **정합성 게이트**를 통과한 집합이어야 한다.

## 4. 해소의 모양 (방향일 뿐, 결정 아님)

둘 중 택일이 아니라 **2계층**으로 보인다:

- **경계 레코드** = 불변·독립 버전의 **진실 원천**(continuously deployed되는 컴포넌트).
- **전역 릴리스** = 그 레코드들을 핀한 **매니페스트/락파일** + 공유 노드 버전 + 모델 선택 + **정합성 보증**.
  git의 (blob들 + tag)와 같되, 지질학 특유의 **coherence 게이트**를 통과한 것.

핵심은 "경계별이냐 전역이냐"가 아니라, **핀된 경계 집합을 유효한 차트로 바꿔주는 정합성 검사를 어떻게 정의하느냐**.

## 5. 열린 질문 (정리)

1. 전역 릴리스 = **복사 스냅샷**인가 **매니페스트/락파일**인가.
2. 정합성 게이트를 **검증(validation)** 수준으로 둘지, **공동추정(joint inference)** 까지 갈지.
3. **단조 순서 불변식**을 hard constraint로 강제할지, lint/경고로 둘지 (대표 경계엔 드물지만 세밀 구간엔 필요).
4. **공유 노드 재보정** 같은 전역 이벤트가 다수 경계 버전을 한꺼번에 bump하는 것을 어떻게 표현·기록할지.
5. **토폴로지/집합 변경**(경계 추가·삭제·개명)의 버전을 어디에 둘지 (경계 레코드 밖, 전역 층?).
6. **샌드박스 오버라이드**(베이스라인 + 경계별 델타)를 스키마로 어떻게 표현할지.
7. **인용**이 (경계 버전 + 전역 릴리스)를 함께 가리키게 하는 최소 형식.
8. **상관된 불확실성**(공유 오차 구조)을 릴리스가 어떻게 보존/전달할지.
